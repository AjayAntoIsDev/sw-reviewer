Use this prompt when judging README quality:

Review the README as documentation for a real user, not as a writing sample. Determine whether it gives enough information for someone to understand the project and use it correctly.

Check for:

- What the project is
- What the project does
- How to access, install, or run it
- How to use its key features
- Any setup requirements
- Any links to demos, releases, stores, package pages, or live deployments

Use stricter expectations for more complex projects:

- Portfolio or simple showcase site: short README can be acceptable if it still explains the project and features.
- CLI, API, libraries, bots, desktop apps, and hardware: expect detailed setup and usage instructions.

Reject the README if it is:

- One line or nearly empty
- Too generic to be useful
- Missing essential install or usage information
- Clearly padded but not informative
- Missing core explanation of the project

If the README looks AI-generated, only use that as a rejection reason if the actual content is low quality, generic, or insufficient.
