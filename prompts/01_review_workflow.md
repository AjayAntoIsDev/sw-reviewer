Follow this workflow for every review:

1. Identify the project type.
2. Open the repository and read the README fully.
3. Confirm the project is open-source and the relevant files are present.
4. Find the demo, binary, package page, store listing, or release artifact required for that project type.
5. Test the project in the way a real reviewer should test it.
6. Check whether the README explains what the project is, how to use it, and any setup or feature details that a user would need.
7. Apply the project-type-specific rules.
8. Decide approve or reject.
9. Write a concise explanation with evidence.

When reviewing:

- Prefer direct verification over assumptions.
- If a link is broken, inaccessible, local-only, or clearly unusable, treat that as failing the demo or functionality requirement unless another valid demo path exists.
- If a requirement is optional, note it as feedback instead of using it as the main rejection reason.
- If the project is an updated project, explicitly mark it as updated.

Minimum evidence to collect before deciding:

- README quality
- Open-source availability
- Working demo or installable artifact
- Project-type-specific shipping requirement
- Any authentication flow if applicable
