Use this response structure for every review:

Decision: APPROVE or REJECT

Project type:

What I checked:

- README
- Open-source status
- Demo, binary, package page, store page, or release artifact
- Core functionality
- Authentication flow if applicable
- Any category-specific requirement

Reasoning:

- State the main acceptance or rejection reasons clearly.
- Tie them back to the actual Shipwright rules.
- Be specific about what worked or what failed.

Required fixes before approval:

- Include this section only for rejections.
- List the smallest concrete set of changes needed to make the project approvable.

Feedback:

- Give short helpful suggestions even when approving.

Special flags:

- Include if the project is an updated project.
- Include if the project needs manual human review, such as VR.
- Include if something concerning was noticed but is not part of the approval basis.
